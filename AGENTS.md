## Build and development commands

Run all commands from the repository root. The project uses Bun:

```bash
# Install dependencies
bun install --frozen-lockfile

# Generate Cloudflare worker types (creates worker-configuration.d.ts)
bun run gen

# Start the Vite dev server
bun dev

# Type-check and build for production
bun run build          # currently fails because src/app.html is missing

# Preview the production build locally via Wrangler
bun run preview

# Type-check only
bun run check

# Run unit tests
bun run test:unit

# Run E2E tests
bun run test:e2e
```

`package.json` scripts:

- `dev`: `vite dev`
- `build`: `wrangler types --check && vite build`
- `preview`: `wrangler pages dev .svelte-kit/cloudflare --port 4173`
- `prepare`: `svelte-kit sync || echo ''`
- `check`: `wrangler types --check && svelte-kit sync && svelte-check --tsconfig ./tsconfig.json`
- `check:watch`: `svelte-kit sync && svelte-check --tsconfig ./tsconfig.json --watch`
- `test:e2e`: `playwright install && playwright test`
- `test`: `bun run test:unit:coverage && bun run test:e2e`
- `gen`: `wrangler types`
- `test:unit`: `vitest`

Important build details:

- The Cloudflare adapter emits to `.svelte-kit/cloudflare`.
- `wrangler.jsonc` sets `pages_build_output_dir` to `.svelte-kit/cloudflare`.
- `tsconfig.json` extends `./.svelte-kit/tsconfig.json` and expects
  `./worker-configuration.d.ts` (generated by `bun run gen`).

## Code organization

Once the application is created, SvelteKit conventions apply:

- `src/app.html` — page template (required, currently missing)
- `src/routes/` — file-system routes (`+page.svelte`, `+layout.svelte`, etc.)
- `src/lib/` — shared library code, importable via `$lib/...`
- `src/lib/server/` — server-only code (must not be imported by client code)
- `static/` — static assets copied to the build output

`vite.config.ts` enforces Svelte 5 runes mode for every project file outside
`node_modules`. The `$lib` alias is provided by SvelteKit.

## Testing strategy

Tests are configured but none exist yet.

**Unit tests (`vite.config.ts`):**

Two Vitest projects are defined:

1. **client** — browser-playwright, Chromium headless
   - Includes: `src/**/*.svelte.{test,spec}.{js,ts}`
   - Excludes: `src/lib/server/**`

   - Includes: `src/**/*.{test,spec}.{js,ts}`
   - Excludes: `src/**/*.svelte.{test,spec}.{js,ts}`

**E2E tests (`playwright.config.ts`):**

- Test files match `**/*.e2e.{ts,js}`.
- Playwright starts a web server with `bun run build && bun run preview` and
  expects the site on port `4173`.

Because `src/` is missing, both unit and E2E test suites will fail or be empty
until the application is scaffolded.

## CI / deployment

`.github/workflows/ci.yml` defines a three-stage pipeline:

1. **Audit** — dependency review on PRs + Trivy filesystem secret/misconfig scan
2. **Lint** — format checks, oxlint, type check, CSS/Svelte/Markdown lint,
   unused-code check (`knip`), and spell check
3. **Build & Test** — build, upload artifact, unit tests with coverage
4. **Deploy** — Cloudflare Pages deploy (only when secrets/vars are present)

⚠️ The CI workflow currently uses `working-directory: portal` and expects a build
output at `portal/target/build`. This does **not** match the repository contents:
there is no `portal/` directory, and the active adapter outputs to
`.svelte-kit/cloudflare`. The workflow must be updated before CI can pass.

Required Cloudflare deployment secrets/variables:

- `CLOUDFLARE_API_TOKEN` (secret)
- `CLOUDFLARE_ACCOUNT_ID` (secret)
- `CF_PAGES_PROJECT` (variable)

## Security considerations

- No secrets, tokens, or credentials are stored in the repository.
- `.gitignore` excludes `.env`, `.env.*`, build output, and `.svelte-kit`.
- Cloudflare credentials are expected to come from GitHub repository secrets/vars.
- Trivy scans for secrets and misconfigurations in CI.
- `worker-configuration.d.ts` is generated by Wrangler and should not be hand-edited.
- Avoid adding literal API keys, passwords, or session identifiers to any source
  file or agent instruction.

## Tooling quirks

- `bun run build` and `bun run check` call `wrangler types --check` first, which
  requires `worker-configuration.d.ts` to exist. Generate it with `bun run gen`.
- The Playwright webServer command in `playwright.config.ts` uses `bun run build`
  and `bun run preview`, matching the Bun-driven project setup.
- Do not run `npm`/`bun` commands from a non-existent `portal/` directory. The
  repository root is the correct working directory.
